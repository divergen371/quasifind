# Design Note: Search Engine Evolution

## 背景

現在の `quasifind` は、ファイルごとに軽量な KMP/BMH ハイブリッドエンジン (`ac_lite`) を構築して検索を行っています。これは単一キーワード検索において「構築コスト vs 検索速度」のバランスが最適ですが、将来的に複数のキーワードを同時に探す場合や、更なる極限の速度を求める場合は、Ripgrep が採用しているような「永続化された重装エンジン」への進化が必要です。

## 次のステップ：永続化エンジン (Persistent AC Engine)

### 1. エンジンの生存期間 (Lifecycle) の変更

現在は C のスタック上にエンジンを作っていますが、これを OCaml のヒープ外（C の `malloc`）で管理し、複数のファイル検索で使い回します。

- **仕組み**:
  1. OCaml 側から検索クエリを受け取った際、最初に 1 回だけエンジンを構築。
  2. エンジンのポインタを OCaml の `Custom_tag` ブロックとしてラップして返す。
  3. 各ファイルの検索時には、そのポインタを渡すだけで即座にスキャン。
  4. OCaml の GC がそのブロックを回収する際に、C 側のメモリも解放する。

### 2. Teddy SIMD アルゴリズムの導入

リテラル（固定文字列）の複数同時検索を加速させるため、Ripgrep の `aho-corasick` クレートが採用している **Teddy** を実装します。

- **原理**:
  - NEON の `vld1q_u8` (16バイト読み込み) と `vtbl1_u8` (テーブルルックアップ) やビットマスク操作を組み合わせ、最大 16 個のキーワードの「アタリ」を 16 バイト単位で同時に判定。
  - 「アタリ」があった場所だけ、DFA（自動機械）を回して詳細を確認する。

### 3. Lazy DFA (怠惰な構築)

巨大な Aho-Corasick テーブル（256バイト × 状態数）の初期化コストを避けるための必須技術です。

- **原理**:
  - 最初は遷移先が「未確定」な最小限のテーブルで開始。
  - 検索中に未知の入力に遭遇した時だけ、その場で遷移（State）を計算してテーブルを埋める。
  - 多くの検索ではパターンのごく一部の状態しか使われないため、メモリと CPU 時間の大幅な節約になる。

## 期待される効果

- **複数パターン検索**: キーワードが 100 個あっても検索速度がほとんど落ちない。
- **スループット**: 起動直後の構築コストがさらに下がり、大規模プロジェクトでの rg との差（現在は 1.2x 程度）を 1.1x 未満に縮められる。
