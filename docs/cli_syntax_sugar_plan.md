# quasifind CLI シンタックスシュガープラン (将来構想)

## 概要

現在の `quasifind` のコア検索エンジンは、型付きで強力な DSL (Domain Specific Language) に依存しています。
これは外部プログラム（GUI ファイルマネージャーのバックエンドなど）から機械的にクエリを組み立てて渡す分には最適ですが、人間がターミナル上で手動で叩く CLI ツールとしては、クォーテーションのネストや厳格なパース規則が原因で非常に使いづらい（認知負荷が高い）という課題があります。

本ドキュメントは、将来的に `quasifind` を一般的な `find` や `fd` のような「人間にとって使いやすい CLI ツール」として進化させるための、シンタックスシュガー（糖衣構文）オプションの導入プランをまとめたものです。
ファイルマネージャー（GUI）の要件とは切り離し、純粋な CLI 体験向上のためのバックログとして記録します。

## 課題

1. **クォーテーションの煩雑さ**: `quasifind . 'name =~ ".*\.txt"'` のように、シェルと DSL の両方でエスケープや引用符を考慮する必要がある。
2. **既存ツールの筋肉記憶とのズレ**: `find . -name "*.txt"` といった既存ツールに慣れきったエンジニアにとって、演算子 (`=~`, `==`) ベースの入力は直感的ではない。
3. **引数とパースの厳格さ**: `[DIR] [EXPR]` の順序や空白の扱いに少しでも違反すると即座に Syntax Error となり、ターミナル上での試行錯誤コストが高い。

## 提案する解決策: オプションフラグによる DSL 自動変換

コマンドライン引数として標準的な Unix 検索オプション（`-name`, `-type`, `-size` など）を CLI フレームワーク (`Cmdliner`) 側で受け取り、プログラム内部でそれらを透過的に DSL 文字列に自動変換してからコアエンジン（AST パージング）に渡す仕組みを構築します。

### 実装の青写真

ユーザー入力が以下のようなオプションベースで行われた場合、内部のクエリビルダーが DSL に翻訳し結合します。

#### 1. 名前検索 (`-name`, `-iname`)

- **ユーザー入力**: `quasifind . -name "*.txt"`
- **内部変換 (DSL)**: `name =~ ".*\.txt$"` (シェルワイルドカードを正規表現に自動変換)
- **ユーザー入力**: `quasifind . -iname "secret"`
- **内部変換 (DSL)**: `name =~ "/(?i).*secret.*/"` (大文字小文字無視の正規表現に変換)

#### 2. ファイル種類 (`-type`)

- **ユーザー入力**: `quasifind . -type f`
- **内部変換 (DSL)**: `type == file`
- **ユーザー入力**: `quasifind . -type d`
- **内部変換 (DSL)**: `type == dir`

#### 3. サイズ指定 (`-size`)

- **ユーザー入力**: `quasifind . -size +100MB`
- **内部変換 (DSL)**: `size > 100MB`
- **ユーザー入力**: `quasifind . -size -1KB`
- **内部変換 (DSL)**: `size < 1KB`

#### 4. 更新日時 (`-mtime`)

- **ユーザー入力**: `quasifind . -mtime -7d`
- **内部変換 (DSL)**: `mtime < 7d` (7日以内)
- **ユーザー入力**: `quasifind . -mtime +1M`
- **内部変換 (DSL)**: `mtime > 1M` (1ヶ月以上前)

#### 5. 複数条件の暗黙の AND 結合

既存の `find` コマンドと同様に、複数のフラグが指定された場合は内部で自動的に `&&` (AND) で結合します。

- **ユーザー入力**: `quasifind . -type f -name "*.ml" -size +10KB`
- **内部変換 (DSL)**: `type == file && name =~ ".*\.ml$" && size > 10KB`

## アーキテクチャ上の利点

- **コアエンジンの変更不要**: AST パーサーや並列走査エンジン (`search.ml`, `eval.ml`) は既存の強力で堅牢なコードをそのまま流用できます。
- **Cmdliner レイヤーでの吸収**: `CLI` エントリポイントである `main.ml` の引数パース部分に「オプショナルなフラグ群」を追加し、それらが指定された場合のみ動的に DSL 文字列を組み上げる薄いラッパー関数を挟むだけで実現可能です。
- **ハイブリッドな利用**: 人間は使い慣れたフラグ (`-name` など) で呼び出し、複雑なクエリが必要な場合や GUI アプリからの RPC 呼び出し時では、生の強力な DSL (`[EXPR]`) を直接渡すという、両方のユースケースをシームレスにサポートできます。

## 今後のステップ

このシンタックスシュガー層を実装することで、`quasifind` は「強力なバックエンドエンジン」という顔と、「人間にとって最高に使いやすい CLI ツール」という二つの顔を両立できるようになります。
現在は「AI 時代の GUI ファイルマネージャーのバックエンド」としての役割を最優先とするため、本件は GUI 開発が一段落した後の CLI UX 向上のマイルストーン（技術的負債・改善案）として着手することを推奨します。
