# Cスタブ リライト設計メモ

このドキュメントは、現在OCamlのプロジェクトで使用しているCスタブ（`stealth_stubs.c`、`search_stubs.c` など）を、メモリ安全性や保守性の観点から他のプログラミング言語でリライトする場合の比較検討と設計方針をまとめたものです。

## 背景と課題設定

現在のCスタブには以下の課題があります。

- OSの低レベルAPIや `argv` のメモリ領域を直接操作している（特にプロセスの隠蔽処理など）。
- `memset` や `memcpy` による文字列長の計算を手動で行っており、バッファオーバーラン等のメモリ破壊リスクが高い。
- OCamlのマクロ（`CAMLparam`、`CAMLreturn` など）を用いたガベージコレクション（GC）ルールの徹底をプログラマ自身が行わなければならない。

これらの課題を解決し、**「OCamlのC API互換性を保ちつつ、メモリ安全性を高める」** ことを目的として、C言語以外の代替言語へのリライトを検討しました。

## 言語比較

リライトの候補となる言語の比較一覧です。

| 言語                 | C互換性 / OCaml連携                                        | Dune導入難易度                                      | メモリ安全性                                                  | 特徴・総評                                                                                                                                           |
| :------------------- | :--------------------------------------------------------- | :-------------------------------------------------- | :------------------------------------------------------------ | :--------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Zig**              | **極めて高い**<br>（Cヘッダをそのまま読込可）              | **比較的容易**<br>（`zig cc` を利用）               | **高い**<br>（スライス、NULLチェック強制）                    | **【C言語のモダンな代替として最適】**<br>Cの書き心地と手軽さを保ちつつ、ポインタ事故を防ぐ。OCaml側のコード変更が不要で即導入可能。                  |

## 各代替言語のメリット・デメリット詳細

### 1. Zig

- **メリット:**
  - `clang` の代替として機能する `zig cc` をDuneから呼び出すだけで済む。
  - `<caml/mlvalues.h>` などのCヘッダを無変換で `import` でき、OCaml側の `external` 宣言も変更不要。
  - クロスコンパイル環境が標準で手に入る。
- **デメリット:**
  - ガベージコレクションがないため、メモリの解放は手動で行う必要がある（`defer` 等）。
  - 言語仕様がv1.0未達であり、破壊的変更の懸念がある。

### 2. Rust

- **メリット:**
  - `ocaml-rs` クレートにより、OCamlのGCとのやり取りにおける型安全性が保証され、不可解なクラッシュを防げる。
  - 並行処理やメモリ管理において最も安全性が高い。
- **デメリット:**
  - OCamlのビルドツール（Dune）とは別にCargoの管理が必要となり、ビルド統合のスクリプト記述が複雑。
  - CのAPIを素直に直訳するとかえって `unsafe` ブロックだらけになり、設計の見直しが必要になる場合がある。

### 3. C++ (Modern C++)

- **メリット:**
  - 現状のDuneの `foreign_stubs` の設定をそのまま流用でき、インフラの変更コストがほぼゼロ。
  - 手書きの文字列・メモリ操作を `std::string_view` や `std::span` 等に置き換えるだけで一定の安全性を確保できる。
- **デメリット:**
  - スマートポインタ等の規律を守らないと、C言語同様の未定義動作の罠に陥るリスクが残る。

## 設計方針・結論

今後の開発リソースと求める安全性のレベルに応じて、以下のいずれかのアプローチを選択することを推奨します。

1. **「安全性を極限まで高めたい・モダンなエコシステムに乗りたい」場合 → 【Rust】**
   - 導入コストは高いが、`ocaml-rs` による恩恵で長期的な保守性とクラッシュ耐性が最大となる。
2. **「Duneのまま、Cの危険なポインタ操作をサクッと減らしたい」場合 → 【C++】**
   - 既存の知識とツールチェーンのまま最も現実的に移行できる。
3. **「Rustは重いが、Cよりは安全・直感的に書きたい」場合 → 【Zig】**
   - 新しいツールを試す意欲があり、かつ C の手軽さを残したい場合に最適。

## PoC 試行記録: Rust 移行 → 不可判定 (2026-02-28)

上記の設計方針に基づき、最高優先候補である **Rust (`ocaml-rs` クレート)** によるビルドシステム統合の PoC を実際に試みた。結果、以下の2点により **現時点での導入は不可** と判断し、`rust-rewrite` ブランチを破棄して `main` に差し戻した。

### 障壁1: `ocaml-boxroot-sys` が OCaml 5.4.0 に非対応

`ocaml-rs` v1.3.0 はデフォルトで `ocaml-boxroot-sys` に依存する。このクレートのビルドスクリプトが OCaml 5 の内部ヘッダ `caml/compact.h` を要求するが、OCaml 5.4.0 のインストール環境（OPAM）にそのヘッダが存在しないためコンパイルが失敗する。

```
cargo:warning=vendor/boxroot/boxroot.c:23:10: fatal error: 'caml/compact.h' file not found
```

`default-features = false` + `features = ["ocaml5", "derive", "no-caml-startup"]` の組み合わせを試みたが、 `ocaml-rs` 1.x の内部で `boxroot` シンボル（`_boxroot_create`, `_boxroot_setup` 等）が依然としてリンク対象に含まれ、リンクエラーが解消されなかった。

`ocaml-sys`（低レベル raw バインディング）に切り替えることでコンパイル自体は通るが、次の障壁に突き当たる。

### 障壁2: Dune の `(foreign_archives)` が静的ライブラリのみの提供に非対応

Dune の `(foreign_archives <name>)` スタンザは、実行時のバイトコード解釈用に **静的ライブラリ (`lib<name>.a`) と動的ライブラリ (`dll<name>.so`) の両方** を常に要求する設計になっている。

Rust の `crate-type = ["staticlib"]` は `.a` のみを生成するため、Dune が `lib/dllrust_stubs.so` を探し出してエラーになる。`cdylib` を追加して `.dylib` も生成しようとすると、今度は `_caml_alloc_initialized_string` 等の OCaml ランタイムシンボルが `cdylib` のリンク時に未定義になる（これらのシンボルは OCaml ホストプロセスが起動した後に初めて解決される性質のもの）。

`(foreign_archives)` を `(c_library_flags -L. -lrust_stubs)` で代替する方法も試みたが、Dune の依存追跡との兼ね合いで完全な解決には至らなかった。

### 結論と今後の方針

| 項目                     | 状況                       |
| :----------------------- | :------------------------- |
| `ocaml-rs` + OCaml 5.4.0 | boxroot 非対応のため NG    |
| `ocaml-sys` + staticlib  | Dune の `.so` 要求で NG    |
| Dune-Cargo 統合の成熟度  | 現時点では公式サポートなし |

**現状**: 今回の P0〜P4 改修によって C スタブは既に十分安全な状態にあるため、現時点では C で継続することが最善。

**再評価のトリガー**:

- `dune-cargo` 等のオフィシャルな Dune/Cargo 統合プラグインがリリースされた場合。
- `ocaml-rs` が OCaml 5 の boxroot なしビルドを公式サポートした場合。

## PoC 試行記録: Zig 移行 → 可能判定 (2026-02-28)

Rust での失敗を踏まえ、**Zig** によるビルドシステム統合の PoC を実施した。結果、Rust での課題をすべて解消し、**実用可能である** と判断した。

### 成功要因1: OCaml 互換のオブジェクトファイル生成の容易さ

Zig は標準で `zig build-obj` および `zig cc` を備えており、OCaml の C ヘッダを `@cImport` で直接読み取ることができる。Rust のように外部クレート (`ocaml-rs`) の OCaml バージョン対応に振り回されることがなく、常にインストールされている OCaml 本体のヘッダを参照するため、バージョン間の不整合が発生しない。

### 成功要因2: macOS 固有のリンク問題の解決

Dune の `(foreign_archives)` が要求する「静的ライブラリと動的ライブラリの両方」の提供について、Zig では以下の3ステップを単一の「Dune ルール」として記述することで解決した。

1.  `zig build-obj`: Zig ソースをオブジェクトファイル (`.o`) にコンパイル。
2.  `ar rcs`: オブジェクトファイルを静的ライブラリ (`.a`) にアーカイブ。
3.  `zig cc -shared`: オブジェクトファイルを動的ライブラリ (`.so`) にリンク。この際、macOS 特有の `-Wl,-undefined,dynamic_lookup` フラグを渡すことで、OCaml ランタイムシンボルを動的に解決可能にした。

これにより、OCaml 側からは従来の C スタブと全く同じインターフェース (`external`) で Zig の関数を呼び出すことに成功した。

### 結論

Zig への移行は技術的に確立されたため、今後は本番スタブの移植へと移行する。

| 項目           | 状況                                                       |
| :------------- | :--------------------------------------------------------- |
| ビルド統合     | 独自 Rule により Dune と完全統合済み (成功)                |
| OCaml 5 互換性 | 標準ヘッダ直接参照により完全対応 (成功)                    |
| 安全性         | スライス、NULLチェック、エラーハンドリングにより向上       |
| 保守性         | Cargo などの外部ツールが不要で OCaml/opam 環境に閉じている |
