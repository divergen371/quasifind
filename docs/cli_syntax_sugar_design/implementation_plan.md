# Quasifind CLI シンタックスシュガー設計 (人間工学・認知心理学アプローチ)

本ドキュメントは、「強力だが認知負荷が高い DSL (Ast.Typed.expr)」を、ターミナルで人間が直感的に扱えるフラグ群に変換するための設計仕様です。人間工学（エルゴノミクス）と認知心理学の観点に基づいて CLI の仕様を定義します。

> [!IMPORTANT]
> ## User Review Required
> この設計方針で実装へと進めて良いか、ユーザーにフィードバックを求めます。
> 特に「サイズ指定時や中身検索時に暗黙的に `type == file` を付加する」という意図推論の仕様について問題がないかご確認ください。

## 1. 設計原則 (Cognitive Psychology Principles)

### 1-1. 最小驚愕の原則と転移学習 (Principle of Least Astonishment & Transfer Learning)
既存の Unix ツール (`find`, `fd`, `rg`) の筋肉記憶をそのまま活かせるようにします。新しい独自のフラグを覚える「再学習コスト」をゼロに近づけます。

### 1-2. シェルとの衝突回避とエラー寛容性 (Error Prevention & Forgiveness)
前回の課題で浮き彫りになった「シェルのリダイレクト (`>`, `<`) に喰われる問題」を構造的に排除します。ユーザーがクォーテーションを忘れても安全に動く（あるいは分かりやすくエラーになる）文法を採用します。

### 1-3. ワーキングメモリの解放と段階的開示 (Chunking)
人間が一度に処理できる情報量には限界があります。`type == file && size > 1MB && name =~ ".*\.txt"` のような1つの巨大な論理式（チャンク）を組み立てさせるのではなく、個別の独立したフラグ (`-t f`, `-s +1MB`, `-e txt`) として記憶・入力できるようにします。

### 1-4. ユーザーの真の意図の推論 (Defaulting to Intent)
「サイズ検索をしているということは、ディレクトリのメタデータサイズではなくファイルを探している」という人間の暗黙の意図をシステム側で先回りして解釈し、本来あるべきクエリへと自動補正します。

## 2. 提案する CLI インターフェース (フラグ仕様)

### 2-1. 名前と拡張子 (名前解決の直感化)
- **`-n, --name <PATTERN>`**
  - **仕様**: GLOBパターンとして処理（`*` や `?` が使える）。
  - **変換先**: `name =~ "GLOB_TO_REGEX"` (内部で正規表現に自動エスケープ変換)
  - **認知効果**: `find -name` と同じ筋肉記憶を利用。正規表現特有のメタ文字エスケープを考える頭の切り替えを不要にする。
- **`-i, --iname <PATTERN>`**
  - **仕様**: 大文字小文字を無視するGLOB。
  - **変換先**: `name =~ "/(?i)GLOB_TO_REGEX/"`
- **`-e, --ext <EXT>`**
  - **仕様**: 拡張子にマッチする。ドットは付けても付けなくても良い (`txt` でも `.txt` でも同じ)。
  - **認知効果**: 拡張子のタイピング時の揺らぎを吸収する。寛容なUI。
  - **変換先**: `name =~ ".*\.ext$"`

### 2-2. 対象の絞り込み (意図推論によるノイズ排除)
- **`-t, --type <TYPE>`**
  - **仕様**: `f` (file), `d` (dir), `l` (symlink)。
  - **変換先**: `type == file` など。
- **`-s, --size <SIZE>`**
  - **仕様**: `+1MB` (1MBより大きい), `-1KB` (1KBより小さい), `10MB` (指定数値と完全に一致)。
  - **意図推論**: **このフラグが使われた場合、暗黙的に `type == file` を AND 条件として結合する。**（対象がディレクトリであることによる検索の汚染を構造的に排除）。
  - **認知効果**: ユーザーはシェルのリダイレクト記号(`<`, `>`)をタイピングするリスクを負わずに、安全な符号(`+`, `-`)でサイズを表現できる。
  - **変換先**: `(type == file && size > 1MB)`
- **`-c, --content <REGEX>`**
  - **仕様**: ファイルの中身自体を正規表現で検索。
  - **意図推論**: これも暗黙的に `type == file` を付加。ディレクトリのコンテンツ検索という非論理的な概念に起因するエラーを防ぐ。

### 2-3. 日時検索
- **`-m, --mtime <TIME>`**
  - **仕様**: `-7d` (7日以内), `+1h` (1時間以上前)。
  - **変換先**: `mtime < 7d` / `mtime > 1h`
  - **認知効果**: `find -mtime` に似ているが、`find` の不親切な仕様（`+1` が実質2日以上前を指すなど）を捨て、`quasifind`の明快な時系列単位系（s, m, h, d）をそのまま直感的に使えるようにする。

## 3. 実行モデル（暗黙の AND 結合）

複数のフラグが指定された場合、それらはすべてユーザーにとって最も自然な推論である `&&` (AND) で結合されます。

```bash
quasifind . -t f -e txt -s +1MB
```
上記の人間工学に即した入力は、システム内部のクエリビルダーにより以下のようにコンパイルされます。
`((type == file) && (name =~ ".*\.txt$")) && (size > 1MB)`

---

## 4. 既存の DSL (生クエリ) との共存

このシンタックスシュガーはあくまで「動的なクエリ生成器」として差し込まれるだけであり、既存の実装である「生クエリ（DSL）を位置引数として受け取る挙動」はそのまま残します。
これにより、
- **対話的ターミナル**: シンタックスシュガーを利用 (`-n "*.ml" -s -10KB`)
- **API/GUIアプリからの呼び出し**: 厳格なDSLを利用 (`"size > 10KB && type == dir"`)
という、相反する2つのユースケースをシームレスに両立できます。
