# Quasifind 使用方法ガイド

Quasifind は、`find` の柔軟な探索能力と `grep` の強力なパターンマッチングを統合し、さらに現代的な SIMD 最適化とデーモンキャッシュを追加した次世代の検索ツールです。

---

## 🚀 クイックスタート

最も基本的な使い方は、探索ディレクトリと **「検索式 (Expression)」** を指定することです。

```bash
quasifind . 'name =~ /\.ml$/'
```

このコマンドは、カレントディレクトリから名前が `.ml` で終わるファイルを検索します。

---

## 🧠 検索式（DSL）の書き方

Quasifind の最大の特徴は、論理演算子 (`&&`, `||`, `!`) を使って複雑な条件を直感的に組み合わせられる点にあります。

### 基本的なフィールド

- **`name`**: ファイル名に対するマッチング
- **`path`**: ファイルのフルパスに対するマッチング
- **`size`**: ファイルサイズ（比較演算子 `> < ==` と単位 `K, M, G` が使用可能）
- **`mtime`**: 最終更新時間（比較演算子と単位 `s, m, h, d` が使用可能）
- **`type`**: ファイルタイプ (`file` または `dir`)
- **`content`**: ファイルの内容（リテラル `==` または正規表現 `=~`）

### 演算子の種類

- **`==`**: 完全一致
- **`!=`**: 不一致
- **`=~`**: 正規表現マッチング
- **`> / <`**: 数値（サイズ・日付）の比較
- **`&&`**: 論理積 (AND)
- **`||`**: 論理和 (OR)
- **`!`**: 否定 (NOT)

### 🔍 正規表現の詳細

Quasifind では、`name`, `path`, `content` フィールドで `/pattern/` 形式の正規表現を使用できます。

#### 構文とエンジン

- **構文**: `/.../` で囲んで記述します（例: `/^test_.*\.ml$/`）。
- **エンジン**: 内部的に PCRE 互換のエンジンを使用しており、モダンな正規表現構文（前方参照を除くほぼ全ての機能）をサポートしています。
- **デリミタの脱出**: パターン内に `/` を含める場合は `\/` と記述してください。

#### 対応フィールド

- **`name =~ /regexp/`**: ファイル名（ベース名）にマッチ。
- **`path =~ /regexp/`**: フルパス全体にマッチ。ディレクトリ構造を含めた複雑な絞り込みに有効です。
- **`content =~ /regexp/`**: ファイルの内容にマッチ。

#### ⚡️ パフォーマンス上の注意

- **リテラル (`==`) vs 正規表現 (`=~`)**:
  固定文字列の検索を行う場合、`content == "FIXME"` のようにリテラル演算子を使用すると、**ARM NEON SIMD 命令** を活用した極めて高速なスキャン（`grep` より 3〜7 倍高速）が適用されます。
  正規表現 (`=~`) も高速ですが、特定の文字列を探すだけならリテラルの方が有利です。
- **大文字小文字の区別**:
  デフォルトで大文字小文字を区別します。無視したい場合は正規表現のフラグ（将来サポート予定）や、文字クラス `/[Tt][Oo][Dd][Oo]/` を活用してください。
- **コンカレンシーと並列処理の最適化**:
  最新のバージョンでは、コンテンツ検索（`content ==` や `content =~`）を実行する C言語スタブ層で OCaml ドメインのランタイムロックを解放します。これにより、巨大なファイルをスキャンしている最中でも他のファイル検索（Eioファイバー）がストールせず、完全なマルチコア並列動作を実現しています。

## ⚡️ 高速コンテンツ検索 (grep alternative)

Quasifind は内部で **ARM NEON SIMD** 命令と、**KMP (Knuth-Morris-Pratt) / BMH (Boyer-Moore-Horspool) ハイブリッドアルゴリズム** を使用しており、`grep -rl` よりも圧倒的に高速です。

```bash
# TODO を含むファイルを高速スキャン（並列度 8）
quasifind -j 8 . 'content =~ /TODO/'
```

### なぜ Quasifind のコンテンツ検索が便利なのか？

`grep` だけでは難しい「属性フィルタとの組み合わせ」が1コマンドで完結します。

```bash
# 10MB以上で、FIXME を含み、かつ1週間以内に更新された Python ファイル
quasifind . 'size > 10M && content == "FIXME" && mtime < 7d && name =~ /\.py$/'
```

## 🕰 デーモンモード (VFS キャッシュ)

頻繁に検索を行うプロジェクトでは、デーモンモードを起動することでディレクトリ走査の時間をゼロにできます。

1. **デーモンを起動**:
   ```bash
   quasifind daemon start
   ```
2. **通常どおり検索を実行**:
   Quasifind は自動的にデーモンを検知し、OS のファイル変更通知 (FSEvents) と同期されたメモリ上の VFS を使って瞬時に結果を返します。

---

## 🛠 コマンドの実行 (`--exec`)

見つかったファイルに対して、後続の処理を自動化できます。

- **各ファイルごとに実行 (`-x`)**:
  ```bash
  quasifind . 'name =~ /\.tmp$/' -x "rm {}"
  ```
- **全ファイルをまとめて実行 (`-X`)**:
  ```bash
  quasifind . 'name =~ /\.log$/' -X "tar czf all_logs.tar.gz {}"
  ```

---

## 💡 パフォーマンスを最大化するヒント

1. **並列度の調整**:
   CPU コア数に合わせて `-j` オプションを指定してください（例: `-j 8`）。デフォルトはシングルスレッドです。
2. **コンテンツ検索の順序**:
   内部的に `name` や `size` などのメタデータフィルタが優先して適用されます。まずファイル名などで対象を絞り込むように式を書くと、実際に `mmap` して内容を読み込むファイル数が減り、さらに高速になります。
3. **デーモンの活用**:
   数百万ファイルがあるような大規模なリポジトリでは、`daemon start` を強く推奨します。

---
