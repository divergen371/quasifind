# Quasifind をバックエンドとした高機能ファイルマネージャー要件定義書

## 1. 概要

本ドキュメントでは、`quasifind` のコア技術をバックエンド（UI と分離された通信デーモン）として活用し、「Path Finder」や「Commander One」のような、極めてレスポンスが高く機能的なファイルマネージャー（エクスプローラー）を構築するためのアーキテクチャおよび機能要件を定義する。

## 2. 目指すユーザー体験 (UX)

- **ゼロ・レイテンシーのディレクトリ展開**: 何千というファイルが含まれる巨大なディレクトリを開いた際でも、カクつきなく瞬時にメタデータ（アイコン、サイズ、日付等）が表示されること。
- **インクリメンタルな超高速検索**: Finder の検索窓のように、文字を打ち込むたびにシステムのあらゆる場所から一瞬で目的のファイルが絞り込まれて表示されること。
- **リアルタイム同期**: ターミナルなど別のツールからファイルが変更・削除された場合、UI を更新しなくてもリアルタイムに画面へ反映されること。

## 3. コア・アーキテクチャ要件（ハイブリッド・エンドポイント）

ファイルマネージャーは「ディレクトリを開く（展開）」という行為と、「ファイルを探す（検索）」という全く異なる特性のワークロードを持つ。
最高性能を引き出すため、UIアプリ（Electron, Tauri 等）はバックエンドとして常駐する `quasifind` デーモンに対し、以下の2つの特化型エンドポイントを使い分けることとする。

### 3.1 ディレクトリの展開（リスト表示）用 API

ユーザーが特定のフォルダを開き、中身を一覧表示する際に利用するエンドポイント。

- **実装アプローチ**: **バルクスキャン API (`getattrlistbulk`) の直叩き**
- **理由/知見**:
  リスト表示では「特定のディレクトリ直下にある全ディレクトリ/ファイルのメタデータ (名前、サイズ、時刻、権限)」を例外なく100%取得する必要がある。
  ここでは従来の `readdir` + `lstat` のループを行うとカーネル・オーバーヘッドが膨大になるため、macOS の `getattrlistbulk`（または Linux の同等機能）といった OS ネイティブなバルク取得 API を経由して、ディレクトリ内の全メタデータをたった 1 回のシステムコールで吸い上げる。
- **ユースケース**:
  `{"cmd": "list_dir", "path": "/Users/atsushi/Documents"}`

### 3.2 高速ファイル検索用 API

ユーザーが検索窓に条件やファイル名を入力し、ディスク全体から目的のものを探す際に利用するエンドポイント。

- **実装アプローチ**: **Lazy Stat (遅延メタデータ解決) + quasifind 既存の `--interactive` 統合**
- **理由/知見**:
  検索においては「条件に合わない何万件ものファイルのメタデータ」をカーネルに計算させるのは致命的なペナルティとなる（VFSロックとディスクI/Oの無駄遣いになる）。
  したがってバルクスキャンは絶対に避け、最も軽量な `readdir` で名前だけを爆速回収し、名前でフィルタ（絞り込み）をパスした「生き残った少数のファイル」に対してのみ `lstat` を実行するという `quasifind` の基本設計（Lazy Stat）をフル稼働させる。
  さらに特筆すべき点として、**`quasifind` には既に `fzf` ライクな対話型あいまい検索機能 (`--interactive` オプション) が実装済み**である。UIアプリ側は新たにファジー検索アルゴリズムを実装する必要はなく、このデーモン側の既存機能に直接クエリをストリーム通信させるだけで、FinderやSpotlight以上の超高速・高精度なインクリメンタル検索ウィンドウを実現できる。
- **ユースケース**:
  `{"cmd": "search_interactive", "query": "rpt23 pdf"}` （デーモン側の `--interactive` 相当のファジーエンジンを作動）

## 4. 通信プロトコル仕様 (Daemon - UI IPC Protocol)

UI層（Tauri/フロントエンド）とバックエンド（quasifindデーモン）間の通信は、パケットヘッダ（JSON）とデータストリーム（TSVまたはJSONL）のハイブリッドプロトコルを採用し、極限のスループットを実現する。

### 4.1 基本リクエスト・レスポンスフォーマット

UI からデーモンへのリクエストはシンプルな JSON を用いる。通信基盤には Unix Domain Socket または標準入出力を利用する（Tauri Sidecar の制約を考慮）。

#### [要件1: 瞬間的なフォルダ展開 (Bulk API)]

UI 側でフォルダを開いたときのリクエスト。

```json
{
  "cmd": "list_dir",
  "path": "/Users/atsushi/Documents",
  "options": { "show_hidden": true, "sort_by": "mtime_desc" }
}
```

**レスポンスストリーム (TSV形式)**: JSONのパース負荷をゼロにするため、タブ区切りでUI（WebView）に直接流し込む。

```text
STATUS: OK\n
NAME\tTYPE\tSIZE\tMTIME\tPERM\n
report.pdf\tfile\t1024500\t1708819200\t644\n
images\tdir\t4096\t1708801200\t755\n
...
```

#### [要件2: インクリメンタル・ファジー検索]

ユーザーが検索窓に `rpt23 pdf` と打ち込んだ瞬間のストリーム通信。

```json
{
  "cmd": "search_interactive",
  "query": "rpt23 pdf",
  "cwd": "/Users/atsushi/",
  "limit": 100
}
```

**レスポンスストリーム (JSONL形式)**: 動的なメタデータやハイライト位置を持たせるため。

```json
{"path": "/Users/atsushi/Documents/report_Q3_2023.pdf", "score": 98, "matches": [18, 19, 20]}
{"path": "/Users/atsushi/Downloads/old_rpt_2023.pdf", "score": 75, "matches": [...]}
```

### 4.2 fsevents / inotify によるリアルタイム同期

デーモンは OS のファイルシステムイベントを監視し、サーバーからの Push 型通知（UI側はリスナー）としてリアルタイムに変更を通知する。

```json
{
  "event": "fs_watch",
  "action": "CREATE",
  "path": "/Users/atsushi/Documents/new_file.txt",
  "meta": { "size": 0, "mtime": ... }
}
```

2. **fsevents (macOS) / inotify (Linux) による監視**
   - デーモンは、現在 UI で開かれている（表示されている）ディレクトリツリーを OS レベルのイベントで監視する。
   - ファイルの追加・削除・変更があった場合、デーモンから UI へ差分更新イベント（Diff）を Push 通知し、UI側で再スキャンをすることなく画面を常に最新に保つ。

## 5. プロフェッショナル向け機能要件 (Inspired by Path Finder / Commander One)

既存の高機能ファイルマネージャーが備える「プロ向け機能」を、高速なバックエンドを活かして再構築する。

1. **デュアルペイン (Dual Pane) とタブUI**
   - Commander One などの代名詞である左右分割ビュー。
   - バルクスキャンAPIの恩恵により、両ペインで数万ファイルのディレクトリを同時に開いてもメインスレッド（UIの描画）をブロックしない。
2. **Drop Stack (一時的スタックエリア)**
   - Path Finder の強力な機能。複数のディレクトリからファイルを一時的にプール（ドラッグ＆ドロップ）し、後で一括して別の場所へ移動・コピーする機能。
   - バックエンドがファイルパスのリストを保持し、OSネイティブ API で一括処理（Move/Copy）を行うことで高速化。
3. **組み込みコンソール (Backend CLI Integration)**
   - フル機能のターミナル・エミュレータを丸ごと載せるのではなく、バックエンドの `quasifind` に対して直接 DSL クエリを印加したり、キャッシュのフラッシュ、プロファイリングなどを実行できる「パワーユーザー / デバッグ向けのコマンドライン・コンソール」をパネルに備える。
   - GUIでは表現しきれない極めて複雑な論理式（例: `name =~ /.*_test\.ml/ && mtime < 2h && size > 10KB`）を直接打ち込んで、結果を即座にUIリストへ反映させることができる。
4. **高度な一括リネーム (Batch Rename)**
   - 正規表現や連番、Exif メタデータ（撮影日など）を利用した高度なリネーム。
   - バックエンドの Zig レイヤーにある高速な正規表現エンジンを活用し、プレビューを数ミリ秒で UI に返す。
5. **非同期ファイル・オペレーションキュー (Operations Queue)**
   - ファイルのコピー・移動・削除などをメインプロセスから切り離し、専用のバックグラウンド・タスクキューで非同期処理する機能。
   - 巨大ファイル移動中も、UI は一切フリーズせずブラウジングを継続可能。並行して走る複数タスクの進捗（プログレスバー）を独立した一つのウィンドウ/パネルで統括表示する。
6. **ディスク使用量ビジュアライザ (Size Browser)**
   - DaisyDisk のような、ディレクトリごとのサイズを算出してツリーマップ / 円グラフで可視化する機能。
   - `quasifind` の `lstat` 高速巡回を用いて、TB（テラバイト）級のディスクのサイズ計算を数秒で完了し、即座に UI にレンダリングする。
7. **Git ステータス直接統合 (Git Integration)**
   - 開発者向け機能。リストビューの各行にそのまま、現在の Git ブランチ、変更状態（M, A, D 等）、および `.gitignore` の反映状態をシームレスに表示する。
8. **内蔵 Hex エディタ / クイックルック (Advanced Preview/Hex Editor)**
   - ファイルを外部アプリで開かずにパネル内で瞬時にプレビューする機構。
   - バイナリファイル選択時は、メモリ効率の良い内蔵 Hex エディタへ自動で切り替える。

## 6. AI 時代に最適化したインテリジェンス要件 (AI-Native Features)

単なる「速い検索」を超え、ファイルシステムと LLM (大規模言語モデル) を結びつける次世代の体験を提供する。

1. **自然言語による曖昧検索 (Semantic Search)**
   - 例: 「先週クライアントから送られてきたPDFの請求書」「去年の夏に沖縄で撮った犬の写真」
   - **実装案**: デーモン（quasifind）がバックグラウンドの低優先度スレッドでファイルのテキスト・メタデータや画像を抽出し、ローカルのベクトル DB (ChromaDB や Qdrant 相当の軽量版) にエンベディングを保存。UI からの自然言語クエリをベクトル検索に変換する。
2. **スマート・オートタギング (AI Auto-tagging)**
   - 画像、PDF、テキストドキュメントを追加した瞬間、デーモンがフックしてローカル AI (Ollama や Llama.cpp など) を用いてコンテンツを要約し、システムタグ (macOS Finder Tags) または内部の拡張属性 (xattr) に自動でタグ付けを行う。
   - これにより、`quasifind` の得意な `Ast.Typed.expr` のメタデータ検索（タグフィルタ）と完璧に連携できる。
3. **Chat with Files (ファイルとの対話)**
   - UI 上でファイル（または複数ファイル）を選択して「AI パネル」を開くと、そのファイル内容をコンテキスト (RAG) として LLM に質問できる。
   - コードファイルを選択して「この関数の説明をコメントでファイルに書き込んで」と指示すれば、デーモンが直接ファイルを書き換える。
4. **Smart Folders (AI駆動型スマートフォルダ)**
   - macOS のスマートフォルダを拡張。「プロジェクトAに関連しそうなファイル」という曖昧なルールを設定しておくと、新しく作られたファイルの内容を AI が解釈し、関連度が高いものを自動で仮想フォルダに集約する。

## 7. GUI 構築の技術選定 (Technology Stack Evaluation)

高パフォーマンスな `quasifind` デーモンと通信するフロントエンド UI を構築するための、主な技術スタックの比較と推奨構成。

### 7.1 候補の比較

| 技術スタック   | 言語 / フレームワーク  | メリット                                                                                    | デメリット                                                                        | 総合適性            |
| :------------- | :--------------------- | :------------------------------------------------------------------------------------------ | :-------------------------------------------------------------------------------- | :------------------ |
| **Electron**   | HTML/CSS/React/Vue     | 圧倒的なエコシステム。VSCode 等での巨大アプリ実績。WebGL(WebGPU)を用いた高度な描画が可能。  | メモリ消費が大きく、バイナリサイズが肥大化しやすい。                              | 🟢 **安定妥協案**   |
| **Tauri (v2)** | Rust (Core) + React等  | **バイナリが超軽量で省メモリ**。OS ネイティブの WebView を使うため高速。IPC通信周りが堅牢。 | IPC のシリアライズに若干の制限や癖がある場合がある。                              | 🚀 **最有力推奨**   |
| **Flutter**    | Dart                   | ピクセルパーフェクトな描画。60fps/120fpsの滑らかなアニメーション。                          | デスクトップ特有のネイティブ感（コンテキストメニュー、D&D）の再現に手間がかかる。 | 🟡 描画特化ならアリ |
| **Native**     | Swift (AppKit/SwiftUI) | 最高の完全な OS 統合。CPU/メモリ効率が最強。                                                | macOS 専売となりマルチプラットフォーム展開が絶望的。                              | 🔴 開発コスト高     |

### 7.2 推奨アーキテクチャ: 【Tauri v2 + React/TypeScript + quasifind】

AI駆動型の次世代ファイルマネージャーを開発するにあたり、**Tauri** を第一推奨とする。

- **理由1: quasifind との相性**
  Tauri はバックエンド（Rust）とフロントエンド（Web）が分離されている。今回の構成では、Rust コア層から更にローカルの `quasifind` デーモンと Unix Domain Socket (または TCP/TSV ストリーム) で通信する仲介役として Tauri を用いることで、Electron のようなメモリ大食いを避けた「極限に軽量な」構成にできる。
- **理由2: UI 表現力と AI 連携**
  React エコシステムを使えるため、AI チャット用の洗練された UI コンポーネント（Vercel AI SDK や Tailwind等）をそのまま流用できる。
- **理由3: ネイティブ・ドラッグ＆ドロップ**
  ファイルマネージャーに必須な高度なシステムインタラクション（ファイルのD&D、コンテキストメニュー）に対して、Tauri v2 では大幅な API の拡充が行われており、十分実用に耐える。

## 8. 結論と次なるステップ

ファイルマネージャー構築においては、今回実験で得られた「**表示用にはバルクスキャン (`getattrlistbulk`) が最強であり、検索用には Lazy Stat が最強である**」という相反するパフォーマンストライアングルを、**デーモン内のエンドポイント分岐（ハイブリッド化）**によって統合することが、最重要の武器となる。

さらに、プロフェッショナル向けの高度なファイル操作機能と、ローカル AI を用いた自然言語での意味検索・分類を統合することで、既存のあらゆるファイルマネージャーを過去のものにする「AI-Native Commander」が実現可能である。
