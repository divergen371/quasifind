# Quasifind をバックエンドとした高機能ファイルマネージャー要件定義書

## 1. 概要

本ドキュメントでは、`quasifind` のコア技術をバックエンド（UI と分離された通信デーモン）として活用し、「Path Finder」や「Commander One」のような、極めてレスポンスが高く機能的なファイルマネージャー（エクスプローラー）を構築するためのアーキテクチャおよび機能要件を定義する。

## 2. 目指すユーザー体験 (UX)

- **ゼロ・レイテンシーのディレクトリ展開**: 何千というファイルが含まれる巨大なディレクトリを開いた際でも、カクつきなく瞬時にメタデータ（アイコン、サイズ、日付等）が表示されること。
- **インクリメンタルな超高速検索**: Finder の検索窓のように、文字を打ち込むたびにシステムのあらゆる場所から一瞬で目的のファイルが絞り込まれて表示されること。
- **リアルタイム同期**: ターミナルなど別のツールからファイルが変更・削除された場合、UI を更新しなくてもリアルタイムに画面へ反映されること。

## 3. コア・アーキテクチャ要件（ハイブリッド・エンドポイント）

ファイルマネージャーは「ディレクトリを開く（展開）」という行為と、「ファイルを探す（検索）」という全く異なる特性のワークロードを持つ。
最高性能を引き出すため、UIアプリ（Electron, Tauri 等）はバックエンドとして常駐する `quasifind` デーモンに対し、以下の2つの特化型エンドポイントを使い分けることとする。

### 3.1 ディレクトリの展開（リスト表示）用 API

ユーザーが特定のフォルダを開き、中身を一覧表示する際に利用するエンドポイント。

- **実装アプローチ**: **バルクスキャン API (`getattrlistbulk`) の直叩き**
- **理由/知見**:
  リスト表示では「特定のディレクトリ直下にある全ディレクトリ/ファイルのメタデータ (名前、サイズ、時刻、権限)」を例外なく100%取得する必要がある。
  ここでは従来の `readdir` + `lstat` のループを行うとカーネル・オーバーヘッドが膨大になるため、macOS の `getattrlistbulk`（または Linux の同等機能）といった OS ネイティブなバルク取得 API を経由して、ディレクトリ内の全メタデータをたった 1 回のシステムコールで吸い上げる。
- **ユースケース**:
  `{"cmd": "list_dir", "path": "/Users/atsushi/Documents"}`

### 3.2 高速ファイル検索用 API

ユーザーが検索窓に条件やファイル名を入力し、ディスク全体から目的のものを探す際に利用するエンドポイント。

- **実装アプローチ**: **Lazy Stat (遅延メタデータ解決) + 並列トラバーサル**
- **理由/知見**:
  検索においては「条件に合わない何万件ものファイルのメタデータ」をカーネルに計算させるのは致命的なペナルティとなる（VFSロックとディスクI/Oの無駄遣いになる）。
  したがってバルクスキャンは絶対に避け、最も軽量な `readdir` で名前だけを爆速回収し、名前でフィルタ（絞り込み）をパスした「生き残った少数のファイル」に対してのみ `lstat` を実行するという `quasifind` の基本設計（Lazy Stat）をフル稼働させる。
- **ユースケース**:
  `{"cmd": "search", "query": "name =~ /\.jpg$/ && size > 1MB"}`

## 4. UI 連携とリアルタイム同期要件

1. **TSV / JSON ストリーミング IPC**
   - バックエンド（quasifindデーモン）から UI への通信は、バッファリングされたストリーミング形式で行う。
   - 膨大な検索結果を返す際は JSON のシリアライズ/デシリアライズコストを削減するため、TSV（タブ区切り）形式の IPC 通信を検討する。

2. **fsevents (macOS) / inotify (Linux) による監視**
   - デーモンは、現在 UI で開かれている（表示されている）ディレクトリツリーを OS レベルのイベントで監視する。
   - ファイルの追加・削除・変更があった場合、デーモンから UI へ差分更新イベント（Diff）を Push 通知し、UI側で再スキャンをすることなく画面を常に最新に保つ。

## 5. 結論と次なるステップ

ファイルマネージャー構築においては、今回実験で得られた「**表示用にはバルクスキャン (`getattrlistbulk`) が最強であり、検索用には Lazy Stat が最強である**」という相反するパフォーマンストライアングルを、**デーモン内のエンドポイント分岐（ハイブリッド化）**によって統合することが、最重要の武器となる。
